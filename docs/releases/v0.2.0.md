# Release Management

## Version: v0.2.0

### Status
- ⏳ Planned

### Objective
> Enable users to edit existing plants and containers, allowing updates to their properties after creation.

### Summary
> This version adds editing capabilities to the core cultivation domain established in v0.1.0.  
> - Allows updating plant properties (name, species, plantedDate, notes)  
> - Allows updating container properties (name, type, location, metadata)  
> - Maintains consistency with existing CQRS architecture and domain events  
> - Provides GraphQL mutations and frontend actions for editing operations

---

## Scope (What's Included)

### Backend
#### Domain
- Plant aggregate: support for updating name, species, plantedDate, notes
- Container aggregate: support for updating name, type, location, metadata
- Validation rules for updates (e.g., cannot edit archived plants)

#### Application / Use Cases
- Update plant
- Update container

#### Infrastructure
- No new infrastructure changes (reuses v0.1.0 setup)

#### Read Model / Projections
- `PlantView`: updated via `PlantUpdated` event projection
- `ContainerView`: updated via `ContainerUpdated` event projection

#### API / Interfaces (GraphQL)
- **Mutations**
  - `updatePlant(id: ID!, input: UpdatePlantInput): Plant`
  - `updateContainer(id: ID!, input: UpdateContainerInput): Container`
- **Queries**
  - `plant(id: ID!): Plant` - New query to fetch single plant for editing

---

### Frontend
#### Views / UI
- Plant detail/edit view (or inline editing in list)
- Container detail/edit view (enhance existing detail view)

#### Actions / Features
- Edit plant (update name, species, plantedDate, notes)
- Edit container (update name, type, location, metadata)
- Form validation and error handling

#### API Integration
- Integrate `updatePlant` GraphQL mutation
- Integrate `updateContainer` GraphQL mutation
- Integrate `plant(id)` GraphQL query

---

## Explicitly Out of Scope (What's NOT Included)

- Bulk editing operations
- Editing relationships (assign/remove handled separately in v0.1.0)
- Editing archived plants
- History/audit trail of changes
- Undo/redo functionality
- Advanced validation rules
- Photo/media editing
- Complex field types or rich text editing

---

## Domain Events

This version emits or introduces the following domain events:

- `PlantUpdated`
- `ContainerUpdated`

---

## Acceptance Criteria

This version is considered complete when:

- [ ] Backend use cases allow updating plant and container properties
- [ ] GraphQL mutations `updatePlant` and `updateContainer` are functional
- [ ] Read models (PlantView, ContainerView) are updated via event projections
- [ ] Frontend allows editing plants and containers through UI
- [ ] Validation prevents editing archived plants
- [ ] Changes are reflected immediately in list and detail views

---

## Technical Notes / Decisions

Relevant architectural or technical decisions taken for this version:

- Reuses existing CQRS architecture from v0.1.0
- Update operations emit domain events for read model synchronization
- Validation rules prevent editing archived plants (business rule)
- GraphQL mutations follow same pattern as v0.1.0 mutations
- Frontend can use inline editing or separate edit view (implementation detail)
